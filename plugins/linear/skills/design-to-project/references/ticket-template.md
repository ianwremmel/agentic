# Ticket Description Template

Use this template for every ticket description generated by the
design-to-project skill. The template is designed to give AI agent implementors
enough context and direction to work autonomously while granting freedom to
explore the codebase and adapt the approach.

## Guidelines for Using This Template

### On "Technical Approach"

This section should provide enough context for the implementor to get started
quickly, but it should NOT be a step-by-step recipe. The implementor is expected
to:

1. Read the referenced design document section
2. Explore the codebase starting from the suggested files
3. Understand existing patterns before modifying code
4. Adapt the approach if they discover a better path

Good: "Extract lambda-specific handlers from `@clc/handlers/src/handlers.mts`
into a new `@clc/handlers-lambda` package. Follow the pattern established by
`@clc/handlers-express` for package structure."

Bad: "1. Create directory `packages/framework/@clc/handlers-lambda`. 2. Create
`package.json` with these exact contents: ..."

### On "Acceptance Criteria"

Every criterion should be independently verifiable. Prefer concrete checks over
subjective assessments.

Good: "`@clc/dispatch` no longer depends on `@aws-sdk/client-sqs`" Bad: "Code is
well-structured"

### On "Dependencies and Coupling"

Be explicit about WHY dependencies exist, not just that they exist. This helps
the implementor understand whether a dependency is truly blocking or if the
approach could be adjusted to remove it.

### On Size Estimates

- **S (Small)**: < 200 lines. Single file or small set of changes.
- **M (Medium)**: 200-500 lines. Multiple files, possibly new test file.
- **L (Large)**: 500-800 lines. Multiple packages, significant new code. At the
  PR budget boundary — may need splitting.
